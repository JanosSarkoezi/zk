So sieht ein Branch aus
```sh

* 4eeb1e6 - bla bla 5 
* 0269faf - bla bla 4
* 82ebe62 - bla bla 3
* c85c844 - bla bla 2
* d5ddbbc - (origin/master, master) bla bla 1
```
Was muss ich machen, wenn ich zoom in master haben moechte. Geht da mit rebase?
Weiterhin sei das auch auf Github vorhanden. Muss man da ein Pullrequest haben
um das zu loesen?

--- 

Das ist eine klassische Git-Situation. Du hast einen Feature-Branch (`zoom`),
der auf dem `master` aufbaut. Um die Änderungen in den `master` zu bekommen,
hast du zwei Hauptwege: **Rebase** oder **Merge**.

Hier ist die Lösung für beide Ansätze, inklusive der GitHub-Perspektive.

### Methode 1: Der saubere Weg mit Rebase (Lineare Historie)

Ein Rebase verschiebt deine neuen Commits (`bla bla 2` bis `6`) direkt an die
Spitze des aktuellen Masters. Da dein `master` laut Log noch auf dem Stand von
`bla bla 1` ist, ist das hier sehr einfach.

1. **Lokal auf den master wechseln:** `git checkout master`
2. **Den zoom-Branch hineinholen:** `git rebase zoom` (Oder `git merge zoom`,
   was in diesem Fall zum selben Ergebnis führt, da es ein "Fast-forward"
wäre).
3. **Auf GitHub aktualisieren:** `git push origin master`

### Methode 2: Über GitHub (Pull Request)

Da dein Projekt auf GitHub liegt, ist der **Pull Request (PR)** der
Standardweg, besonders wenn du in einem Team arbeitest oder Code-Reviews
möchtest.

1. **Push deinen Branch:** (Hast du laut Log schon gemacht: `origin/zoom`).
2. **PR erstellen:** Geh auf GitHub. Dort erscheint meist direkt ein gelber Button "Compare & pull request".
3. **Merge-Option wählen:** Auf GitHub kannst du beim Bestätigen des PRs wählen:
* **Create a merge commit:** Erstellt einen extra Commit, der beide Zweige verbindet.
* **Squash and merge:** Fasst alle 5 "bla bla" Commits zu einem einzigen zusammen (macht den Master sehr sauber).
* **Rebase and merge:** Übernimmt alle Commits einzeln ohne Merge-Commit.

### Muss man einen Pull Request haben?

**Technisch gesehen: Nein.** Du kannst alles lokal auf deinem Rechner mergen
und direkt in den `master` pushen (wie in Methode 1 beschrieben).

**Praktisch gesehen: Ja, meistens.**

* Viele GitHub-Repositorys haben den `master` (oder `main`) **geschützt**
("Protected Branches"). Dann *kannst* du gar nicht direkt pushen und **musst**
einen PR benutzen.
* Ein PR bietet dir die Chance, deine Änderungen noch einmal im Browser zu
prüfen, bevor sie final im Hauptzweig landen.

### Zusammenfassung: Was solltest du jetzt tun?

Wenn du alleine arbeitest und volle Rechte hast:

```sh 
git checkout master
git rebase zoom 
git push origin master

```

Wenn du professionell arbeiten willst oder der Master geschützt ist:

1. Geh auf GitHub.com.
2. Erstelle einen **Pull Request** von `zoom` nach `master`.
3. Klicke auf **Merge**.
